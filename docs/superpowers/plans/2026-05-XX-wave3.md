## 2. Lead-time master list

The single hardest constraint at Wave 3 is **vendor lead time on Paddle AUP review**. This dictates the calendar of any paid launch.

| Spec | External lead time | Internal eng. effort | Sequencing notes |
|---|---|---|---|
| `#b-auth` | 0 days (self-serve Clerk) | 1-2 days | First; gates `#b-multitenant` + `#b-encryption` |
| `#b-multitenant` | 0 days | 2-3 days | Blocks on `#b-auth` |
| `#b-encryption` | 0 days | 5-7 days + 2 days migration testing | Blocks on `#b-auth`; can run parallel with `#b-multitenant` |
| `#b-deploy` | 1-2 days domain + cloud setup | 2-3 days | Independent; can run from day 1 |
| `#b-security-review` | 0 days | 5-7 days walking checklist | Final gate before alpha; recurring weekly post |
| `#b-privacy-policy` | **2-4 weeks lawyer review** (paid launch) / 1 week self-written (alpha) | 1-2 days publication | Critical path for paid launch |
| `#b-paddle` | **2-4 weeks Paddle AUP** + **2 weeks Stripe Atlas LLC** | 5-7 days webhook implementation | **CRITICAL PATH BLOCKER** for paid launch (~6-8 weeks total) |

**Critical path for paid launch: `#b-paddle` LLC formation + AUP review = 6-8 weeks calendar.** All other specs parallelize within this window.

**Critical path for alpha (free) launch: `#b-auth` + `#b-deploy` + `#b-security-review` = ~2 weeks calendar** (no Paddle, no LLC required).

**Recommended phasing:**

- **Wave 3a (alpha, free):** `#b-auth` + `#b-multitenant` (row-level) + `#b-deploy` + `#b-security-review` + self-written `#b-privacy-policy`. Ships in ~2 weeks after Wave 2 dispatch closes.
- **Wave 3b (paid launch):** `#b-paddle` + lawyer-reviewed `#b-privacy-policy` + `#b-encryption`. Ships ~6-8 weeks after Wave 3a, gated on Paddle AUP approval.

This phasing is honest with the brand-book §6 anti-spin posture: alpha ships at-rest plaintext per friends-intro disclosure; paid launch hardens encryption + ships MoR billing.

## 5. Legal exposure summary

### GDPR

EU geo-block at Cloudflare edge (`#b-deploy`) defends Article 3 territorial scope; user-declared residence at signup adds dual-layer. Disclosure: privacy policy explicitly states non-targeting. Right-to-erasure: `#b-multitenant` cascade DELETE on user_id. Data portability: markdown export shipped Wave 1b. **Article 32 technical measures: encryption-at-rest (`#b-encryption`) is post-alpha; alpha ships at-rest plaintext per friends-intro honest disclosure.** Breach notification: 72hr SLA — incident-response runbook needed before alpha launch.

### Sensitive personal data (GDPR Art. 9)

Conversational disclosures in Rodix may contain mental-health / political / religious / sexual-orientation categories. **This is structurally unavoidable** given the product is for "thinking on hard things." Mitigation stack: encryption-at-rest (Wave 3b) + geo-block EU + heightened privacy-policy clauses (lawyer review) + DPO contact + crisis-content handoff protocol (Wave 1c).

### Anthropic ZDR + Crisis content

Per `assumption-list.md` D1: emotional-content extraction is built (V3 GREEN); conversational-system register refuses Caregiver phrasing; crisis-protocol NOT yet implemented. **Wave 1c must ship before Phase 1 alpha launch** — this is a precondition for all of Wave 3, not just `#b-privacy-policy`. Per `#b-security-review` checklist: crisis trigger must bypass extraction (no `topic: suicide` cards ever written), surface resource banner.

Anthropic ZDR (Zero Data Retention) opt-in: confirm Rodix's API tier supports ZDR, or disclose default 30-day retention in privacy policy honestly. **TODO at Wave 3 dispatch:** web_fetch Anthropic's current policy.

### Founder-anonymity vs LLC public records

The strongest tension in Wave 3. Friends-intro: "Solo, anonymous, working out of Asia" is brand-trust positioning. LLC formation creates public records exposing ownership. Wyoming LLC offers shielded member listing; Delaware does not; holding-entity layer adds a nominee shell. Cost: $300-1k/yr ongoing. **TA-Wave3-1 above is the operational choice.** Without LLC, no MoR; without MoR, solo founder bears full US sales-tax + EU VAT exposure (incompatible with §6 Constraints in `position-strategy.md`).

### Audit & disclosure honesty

Brand-book §7 Decision 6 commits to "we never claim zero-knowledge or end-to-end encryption when reality is server-side recall with at-rest plaintext." Privacy policy + marketing copy must hold this line. Lawyer review (`#b-privacy-policy` option b) catches legal non-compliance; brand-voice review catches register drift. Both passes required.

## 6. Phase 1 → Phase 2 / 3 transition

### Phase 1 (alpha launch + paid launch within ~6-8 weeks)

- Wave 3a alpha: `#b-auth` (Clerk) + `#b-multitenant` (row-level) + `#b-deploy` (Railway+CF) + `#b-security-review` (self-audit) + `#b-privacy-policy` (self-written).
- Wave 3b paid: + `#b-paddle` (Paddle MoR after AUP) + `#b-encryption` (SQLCipher) + `#b-privacy-policy` (lawyer-reviewed).
- Device priority: desktop Web (per `project_device_priority.md`).

### Phase 2 (launch+30): PWA install + Tauri desktop

- `#b-auth` extends to PWA scope (Clerk SDK supports). Mobile-responsive web continues to be primary mobile surface.
- `#b-encryption` already shipped; PWA does not change tenancy boundary.
- `#b-deploy` may shift to Hetzner if Railway costs cross threshold during Phase 1.
- New surface: PWA install copy must honor brand-book §5 voice (no "install our app!" sycophancy).

### Phase 3 (launch+90): iOS/Android native

- Apple Developer Program enrollment ($99/yr) — anonymous-founder posture intersects badly; LLC entity required, may need real-name founder in App Store Connect agreement. **Type-A escalation deferred to Phase 3.**
- Native auth via Clerk's iOS/Android SDKs.
- Native encryption: per-platform Keychain / Keystore for key storage; SQLCipher binary builds for ARM64.
- `#b-paddle` mobile in-app purchase: Paddle has limited mobile-IAP support; Apple/Google IAP fees are 15-30% (vs Paddle 5%). Significant margin compression on mobile native; revisit pricing for in-app vs web-purchase parity.

### Wave 4+ (post-launch hardening)

- `#b-multitenant` row-level → per-user-file migration at >5k users (TA-Wave3-13).
- `#b-encryption` user-controlled passphrase mode if alpha telemetry shows demand (TA-Wave3-12).
- `#b-security-review` external pen-test at $10k MRR (TA-Wave3-15).
- Bug-bounty program (HackerOne / Intigriti) at Phase 2.
- SOC 2 / ISO 27001 at $100k MRR.

### What does NOT change in transition

Per brand-book §8 five-year coherence test: the four bets (white-box cards / cross-model / active recall / real markdown export) are architectural commitments, not features. Wave 3 SaaS plumbing must not introduce constraints that contradict any of the four bets. Specifically:

- **Cross-model (Bet 2):** Clerk does not gate any specific AI provider; `#b-paddle` does not bind to a single AI provider. Both honored.
- **Real export (Bet 4):** `#b-multitenant` cascade DELETE cleans tenancy; markdown export stays user-scoped. `#b-encryption` does not encrypt the markdown export at the file level (export remains plaintext per friends-intro: "Open it in Obsidian, paste it into Notion, throw it on a USB stick"). Honored.
- **White-box cards (Bet 1):** No telemetry layer in Wave 3 captures shadow profiles. `#b-multitenant` TA-3 explicitly bans user-scoped behavioral analysis. Honored.
- **Active recall (Bet 3):** `#b-encryption` does NOT achieve zero-knowledge — recall requires plaintext at query time. Disclosed honestly per friends-intro. Honored.
