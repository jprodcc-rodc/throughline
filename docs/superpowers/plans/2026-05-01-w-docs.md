# `#w-docs` — Implementation Plan

> **For agentic workers:** Use `superpowers:test-driven-development` where applicable(docs are markdown,less TDD-able but verifiable).

**Goal:** Write 3-5 core docs pages for alpha launch:Getting started / 历史导入教程 / API key 配置 / 隐私 / FAQ.

**Architecture:** Static markdown files served via marketing site framework(Astro recommended per `marketing-site-design.md`)。If marketing site not yet stood up,host docs as `/docs/*` static pages from `app/web/`(temporary)until `marketing-site-design.md` plan executes.

**Tech Stack:** Markdown · marketing site framework(Astro recommended)or temporary static serving from `app/web/`

## §7.4 5 项 framing

| | |
|---|---|
| Visual | Markdown rendered with marketing site theme(B + Lucide + Inter,继承 visual system). 每页:title + table-of-contents inline + sections + sidebar nav. |
| 产品策略假设 | Alpha 用户 + early launch 用户自助解决问题 = reduce 支持负担 = Rodc 不被 1-on-1 troubleshooting 淹. |
| 适合 / 不适合 | **适合**: alpha + launch+30 阶段(用户少,docs 是主要支持渠道). **不适合**: 100+ 用户量 community-driven era(改用论坛 / Discord). |
| Trade-off | + 一次写好 = launch 后 reduce friction / − 文档跟产品 drift 风险(所以放重点 5 篇,不堆量) |
| 最大风险 | docs 跟产品迭代不同步,一周后过期. **缓解**:每篇文档结尾标 "last updated YYYY-MM-DD" + changelog(per `marketing-site-design.md` `#w-changelog`)+ 5 篇是核心(不是 50 篇),维护成本低. |
